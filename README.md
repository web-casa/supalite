> Built from upstream Supabase images, minus the parts most small

Self-hosted Postgres + REST + Auth + Studio + admin panel, assembled from
upstream Supabase images without the pieces that most small deployments
don't need (Storage, Edge Functions, Realtime).

One command bootstraps the whole stack. Day-2 operations happen through a
Go-powered admin panel baked into the gateway.

---

## Quick start

```bash
./setup.sh
```

`setup.sh` generates all secrets (`POSTGRES_PASSWORD`, `JWT_SECRET`,
`ADMIN_TOKEN`, …), writes `.env` (mode 600), pulls images, starts the stack,
health-checks every public endpoint, and prints the admin token.

Landing page: <http://localhost:8000/> — links to Studio and the admin panel.

Lost the admin token? It's in `.env` under `ADMIN_TOKEN`.

## What's inside

| Service    | Image                          | Role                                                  |
|------------|--------------------------------|-------------------------------------------------------|
| `db`       | supabase/postgres + pgbackrest | PG 15 + 12 Supabase roles + 12 extensions + pgBackRest|
| `rest`     | postgrest/postgrest            | Auto REST API derived from the schema                 |
| `gotrue`   | supabase/gotrue                | Email + OAuth auth, JWT minting                       |
| `admin`    | (built from `./admin`)         | Go backend + Next.js SPA — this project's admin panel |
| `meta`     | supabase/postgres-meta         | Metadata API for Studio (profile: `studio`)           |
| `studio`   | supabase/studio                | Upstream Supabase Studio (profile: `studio`)          |
| `gateway`  | caddy:2-alpine                 | Reverse proxy, CORS, auto-HTTPS, admin-cookie gate    |

Studio + postgres-meta run under `COMPOSE_PROFILES=studio` (default).
Remove that line in `.env` for the minimum stack.

Architecture diagram: [`architecture.mmd`](./architecture.mmd) (Mermaid).

## Admin panel

`http://localhost:8000/admin/` — cookie-auth, HMAC-signed, SameSite=Strict.
Login with the `ADMIN_TOKEN`.

| Page      | What it does                                                   |
|-----------|----------------------------------------------------------------|
| Dashboard | Service status + API keys                                      |
| Logs      | Live SSE tail of any container's stdout/stderr                 |
| DB Ops    | VACUUM / VACUUM FULL / ANALYZE / REINDEX, target-aware          |
| Sessions  | `pg_stat_activity` with `pg_terminate_backend` action           |
| Backups   | pg_dump → S3 (run / list / restore / delete) **+** pgBackRest (init / full / diff / incr / restore) |
| Settings  | Per-section config: General, SMTP (with Send Test), GitHub / Google / Apple OAuth (with Test credentials), Backup S3 |

State transitions worth noting:

- Sidebar service dots update via `/api/status/stream` (SSE), not 30 s polling.
- Long-running ops (pg_dump, pg_restore, pgBackRest backup/restore) run
  async; the UI polls status and pauses polling when the tab is hidden.
- Destructive actions (pg_restore, pgBackRest restore, session terminate,
  backup delete) require a typed-label / typed-name confirmation.

Switch to Studio for application-developer work (table editor, SQL editor,
auth user management): sidebar "Open Studio" link, or `/studio/`.

## Environment variables

All config lives in `.env`. See [`.env.example`](./.env.example) for the
full annotated list. Most fields are editable via the admin Settings page
(writes to `.env`, then needs a restart for the affected service). A
subset is read by the admin process each request (e.g. SMTP test) so
Save-then-Test works without restarts.

Required at bootstrap (auto-generated by `setup.sh`):

- `POSTGRES_PASSWORD` — shared by all 12 Supabase roles
- `JWT_SECRET` — PostgREST + GoTrue signing key
- `ADMIN_TOKEN` — admin panel login
- `COOKIE_SIGNING_KEY` — HMAC for the admin session cookie
- `PG_META_CRYPTO_KEY` — Studio's secret-at-rest

## Enabling HTTPS

Default is http-only on `:8000`. To put Caddy in front with auto-TLS:

1. Point public DNS (A/AAAA) at the host.
2. `.env`: `CADDY_SITE_ADDR=db.example.com` (space-separate for multi-host).
3. `.env`: `API_EXTERNAL_URL=https://db.example.com`.
4. Make ports 80 + 443 reachable from the internet (ACME HTTP-01 needs `:80`).
5. Start with the HTTPS override file:
   ```bash
   docker compose -f docker-compose.yml -f docker-compose.https.yml up -d
   ```

Caddy auto-issues and renews Let's Encrypt certs. State persists in the
`caddy-data` named volume.

The HTTPS override is kept separate from `docker-compose.yml` so a default
`docker compose up` never silently binds host `:80`/`:443` — upgrades never
collide with existing web services on the same host.

Custom cert (internal CA, wildcard)? Edit `volumes/api/Caddyfile` directly
and use the `tls` directive.

## Multiple frontends on one instance

One SupaLite serves a typical indie project just fine — the same
auth backend backing your Next.js web app, your mobile app, and your
marketing site. Each origin needs to be allowed for CORS, and any
redirect URIs different from `SITE_URL` need to be on GoTrue's allow
list. Both are env-driven.

In `.env` (or via the admin panel's Settings → General tab):

```
SITE_URL=https://app.example.com
CORS_ALLOWED_ORIGINS_REGEX=https://app\.example\.com|https://admin\.example\.com|https://marketing\.example\.com
GOTRUE_URI_ALLOW_LIST=https://admin.example.com/auth/callback,myapp://auth
```

- `CORS_ALLOWED_ORIGINS_REGEX` is a regex alternation; escape dots.
- `GOTRUE_URI_ALLOW_LIST` is comma-separated; `SITE_URL` is implicitly
  allowed, so list only the extras. Mobile deep links (`myapp://…`) go
  here.
- Keep `SITE_URL` pointing at your **primary** frontend — it's the
  only value used inside confirmation / magic-link email templates.

Restart the gateway + gotrue after changing these:

```
docker compose up -d gateway gotrue
```

## Multiple unrelated projects on one host

Different product, shared hardware: run one compose project per app.
Each gets its own Postgres data volume, its own admin token, its own
ports.

```bash
# Project A
cp .env .env.a
$EDITOR .env.a         # unique POSTGRES_PASSWORD, ADMIN_TOKEN, etc.
docker compose -p app-a --env-file .env.a up -d

# Project B — bind a different host port for the gateway
cp .env .env.b
$EDITOR .env.b         # also change GATEWAY_HTTP_BIND / port if reusing 8000
GATEWAY_HTTP_BIND=127.0.0.1  # then put 127.0.0.1:8001:8000 in an override
docker compose -p app-b --env-file .env.b -f docker-compose.yml -f compose.b.override.yml up -d
```

Each project's containers are namespaced (`app-a-db-1`, `app-b-db-1`);
data volumes are isolated (`app-a_db-data`, `app-b_db-data`). Route
external traffic with a top-level Caddy/Traefik by domain.

This is the right pattern when projects have **no shared user base**
or you want blast-radius isolation (A compromised doesn't leak B).
For shared-auth-same-product use the single-instance multi-origin
path above.

## Backups

Two independent backup paths target the same S3-compatible storage
(AWS / R2 / MinIO / B2 / …).

### Logical — `pg_dump` to S3 (always available)

- Admin panel → Backups → Run Backup
- Runs `pg_dump -Fc --no-owner --no-acl`, streams directly into S3
  (no temp files, works for arbitrary DB size).
- Restore: click the rewind icon on a backup row, type the filename to
  confirm. Calls `pg_restore --clean --if-exists` against the live DB.
- 30-minute timeout; SIGTERM-then-SIGKILL grace on cancel.

### Physical — pgBackRest (opt-in, incremental)

- Enable in `.env`: `PGBACKREST_ARCHIVE_MODE=on` (plus the other
  `PGBACKREST_*` defaults), then `docker compose up -d --build db`
  (custom image adds ~5 MB of binary).
- Admin panel → Backups → pgBackRest section → Initialize Stanza.
- Run full/diff/incr backups; WAL archives to S3 continuously.
- Restore orchestrates **stop db → one-shot restore container →
  start db**. On failure, db stays stopped so you can inspect before
  bringing the cluster back up.
- `--delta` restore over the existing data directory, so no
  "empty-volume-first" dance.

**Security note:** both paths share `BACKUP_S3_*` credentials. The S3
secret is stored in `.env` and mounted into the db and admin containers
through compose env vars. Anyone with access to `docker.sock` on the
host can read those via `docker inspect`.

## Operational diagnostics

| Need                             | Where                                   |
|----------------------------------|-----------------------------------------|
| Service up/down                  | Sidebar dots / `GET /api/status`        |
| Live log tail                    | Logs page / `GET /api/logs/stream`      |
| Active queries, idle txns        | Sessions page / `GET /api/sessions`     |
| Kill a runaway session           | Sessions row → Terminate (typed PID)    |
| Run maintenance                  | DB Ops page / `POST /api/dbops/maintenance` |
| Test SMTP without registering    | Settings → SMTP → Send Test             |
| Validate OAuth credentials       | Settings → GitHub/Google → Test credentials |
| Restart GoTrue + gateway         | Settings → Restart Services             |

## Development

Admin panel source lives in `./admin`:

- `internal/` — Go packages (handlers, docker client, sse, backup, db, …)
- `web/` — Next.js app (statically exported, embedded via `go:embed`)

Rebuild after edits:

```bash
docker compose up -d --build admin
```

`admin/web/` is a flat Next.js 16 project — see `admin/web/AGENTS.md` for
version-specific gotchas. `go vet ./...` and `go build ./...` must be run
from `admin/` (not `admin/web/`, which has no Go code and silently "passes").

There are no unit tests yet; verification has been Codex-review-and-fix
cycles on each phase.

## Design decisions worth remembering

- **Gateway is the only exposed port.** PostgreSQL binds loopback by
  default (`POSTGRES_BIND_ADDR=127.0.0.1`); flip to `0.0.0.0` only if
  you know why.
- **Studio is optional.** It's one of the best parts of Supabase, but it's
  also the heaviest; removing `studio` from `COMPOSE_PROFILES` drops ~800 MB
  of images without losing any API-level capability.
- **Two separate auth layers.** The Caddy `require_apikey` snippet gates
  `/rest/*` and `/graphql/*` on `ANON_KEY` / `SERVICE_ROLE_KEY`. The admin
  panel uses a separate HMAC cookie signed with `COOKIE_SIGNING_KEY` — a
  compromised `ANON_KEY` can't touch the admin API.
- **Supabase-upstream compat first.** The `db` init script
  (`volumes/db/init/zz-SupaLite.sh`) runs AFTER
  supabase/postgres's own `migrate.sh` and only fills in what the upstream
  image doesn't: 9 extra extensions, role passwords unified to
  `POSTGRES_PASSWORD`, `auth.uid/role/email` functions rewritten for the
  new `request.jwt.claims` GUC mode.
- **Docker socket in admin is deliberate.** The admin container mounts
  `/var/run/docker.sock` so it can `docker compose up --no-deps gotrue
  gateway` on service restart, exec into db for `pg_dump` / pgBackRest,
  and orchestrate pgBackRest restore. The trust model is "admin-panel
  access == root on the host"; anyone holding the admin cookie can
  already write `.env`, so Docker-socket access isn't an additional
  vector.
